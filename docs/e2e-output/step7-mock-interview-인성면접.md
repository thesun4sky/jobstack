## Q1. 팀 프로젝트에서의 갈등 해결

### 면접관 질문

> "자소서에 졸업 프로젝트 '밥친구'에서 팀 리더를 맡으셨다고 했는데, 팀원 간 의견 충돌이 있었던 경험을 이야기해주세요. 어떻게 해결하셨나요?"

### 김민수 시뮬레이션 답변

> "네, 밥친구 프로젝트 초반에 기술 스택을 결정할 때 의견 충돌이 있었습니다. 저는 Spring Boot를 제안했는데, 팀원 중 한 명이 Node.js Express가 더 빠르게 개발할 수 있다고 주장했습니다. 사실 그 친구 말도 맞았는데, 저는 우리가 배운 Java 기반이 안정적이라고 생각했습니다. 그래서 결국 투표를 해서 3:1로 Spring Boot를 선택했습니다. 다행히 나중에 그 팀원도 Spring Boot에 적응하면서 잘 마무리되었습니다."

### 4차원 평가

| 차원 | 점수 (1-5) | 평가 |
|------|:----------:|------|
| 구체성 | 3 | 상황은 구체적이나, "투표로 결정"이라는 해결 방식이 피상적 |
| 논리성 | 2 | 왜 Spring Boot가 더 나은 선택인지 논리적 근거 부족. "우리가 배운 Java 기반"은 약한 논거 |
| 진정성 | 4 | 솔직하고 자연스러운 답변. "그 친구 말도 맞았는데"라는 표현에서 진정성 느껴짐 |
| 직무관련성 | 3 | 기술 스택 선택이라는 개발 관련 갈등이지만, 기술적 깊이가 부족 |

**소계: 12/20**

### 피드백

**잘한 점**:
- 상대방 의견을 인정("그 친구 말도 맞았는데")하는 열린 태도 좋음
- 실제 있었던 일처럼 자연스러움

**개선할 점**:
- "투표로 결정"은 리더십이 아니라 민주주의입니다. 면접관은 **리더로서의 판단과 설득 과정**을 듣고 싶습니다.
- 결론이 "다행히 잘 마무리"로 끝나는데, **본인이 한 구체적 행동**이 빠져 있습니다.

**모범답안 방향**:

> "Node.js의 개발 속도 장점을 인정하면서도, 팀원 4명 중 3명이 Java 경험만 있어 학습 비용이 높다는 점을 **데이터로 정리**했습니다. 'Sprint 1에서 Spring Boot는 3일, Express는 7일(Node.js 학습 포함) 소요 예상'이라는 비교표를 만들어 공유했고, **프로토타입을 2일 안에 각각 만들어 비교하자**고 제안했습니다. Spring Boot 프로토타입이 먼저 동작했고, 그 팀원도 '이 속도면 괜찮겠다'며 합류했습니다. 이 경험에서 **감정이 아니라 데이터와 실험으로 설득하는 것**이 중요하다는 것을 배웠습니다."

### 꼬리질문 예시

1. "투표로 결정했다고 하셨는데, 만약 2:2로 갈렸다면 어떻게 하셨을 것 같나요?"
2. "반대한 팀원이 이후에도 불만을 가지고 있었다면 어떻게 하셨을까요?"
3. "리더로서 '내 의견을 관철하는 것'과 '팀의 합의를 이끌어내는 것' 중 무엇이 더 중요하다고 생각하나요?"

## Q2. 기술적으로 몰입했던 경험

### 면접관 질문

> "네이버 인재상 중 하나가 '몰입'인데요. 기술적으로 가장 깊이 몰입했던 경험을 이야기해주세요."

### 김민수 시뮬레이션 답변

> "인턴 기간에 Redis 캐싱을 적용한 경험이 가장 몰입했던 것 같습니다. API 응답 속도가 느리다는 이슈가 있었는데, 제가 Redis를 적용해보겠다고 자원했습니다. 처음에는 Redis를 써본 적이 없어서 공식 문서와 인프런 강의를 보면서 공부했고, 일주일 정도 걸려서 적용했습니다. 적용하고 나니 응답 속도가 많이 빨라져서 뿌듯했습니다. 팀장님도 좋은 평가를 해주셨습니다."

### 4차원 평가

| 차원 | 점수 (1-5) | 평가 |
|------|:----------:|------|
| 구체성 | 2 | "많이 빨라졌다"는 추상적. 구체적 수치 (350ms→15ms) 언급 없음 |
| 논리성 | 3 | "느리다→Redis 적용→빨라졌다" 흐름은 있으나, 왜 Redis인지 논거 부족 |
| 진정성 | 4 | "처음에 써본 적 없어서 공부했다"는 솔직한 표현 |
| 직무관련성 | 4 | Redis 캐싱은 서버 플랫폼 개발 직무에 직결 |

**소계: 13/20**

### 피드백

**잘한 점**:
- 자발적으로 자원했다는 점(주도성) 언급
- 학습 과정을 솔직하게 이야기

**개선할 점**:
- "몰입"을 보여주려면 **밤새 삽질한 구체적 장면**이 필요합니다. "일주일 걸렸다"만으로는 몰입이 전달되지 않습니다.
- **기술적 디테일**이 빠져 있습니다. 어떤 문제를 만났고, 어떻게 해결했는지.
- "뿌듯했습니다"는 감정형 종결입니다.

**모범답안 방향**:

> "응답 속도 350ms라는 이슈를 받고, 원인 분석부터 시작했습니다. 프로파일링 결과 DB 쿼리 자체는 50ms인데 ORM 직렬화에서 300ms가 소요되고 있었습니다. 그래서 '쿼리 최적화가 아니라 결과 캐싱이 정답'이라고 판단하고 Redis를 제안했습니다. 처음 적용했을 때 캐시 키 설계를 잘못해서 같은 데이터가 다른 키로 중복 저장되는 문제가 생겼습니다. 이틀간 Redis Monitor 명령어로 실시간 키 생성 패턴을 관찰하면서 원인을 찾았고, 키 네이밍 규칙을 `{entity}:{id}:{version}` 형식으로 통일한 후 적중률이 65%에서 92%로 올랐습니다. 그 이틀간 퇴근 후 카페에서 Redis 공식 문서의 Eviction Policy 챕터를 전부 읽었는데, 그때 **'모르는 것을 파고드는 재미'**를 처음 느꼈습니다."

### 꼬리질문 예시

1. "Redis 외에 다른 캐싱 방법(Ehcache, Memcached 등)은 고려하지 않았나요? 왜 Redis를 선택했나요?"
2. "캐시 데이터가 DB와 불일치하는 상황이 발생하면 어떻게 처리하나요?"
3. "그 경험 이후 학습 방법에 변화가 있었나요?"

## Q3. 실패 또는 실수 경험

### 면접관 질문

> "개발하면서 가장 큰 실수를 했거나 실패한 경험이 있다면 이야기해주세요. 그리고 거기서 무엇을 배웠나요?"

### 김민수 시뮬레이션 답변

> "인턴 때 운영 DB에 잘못된 쿼리를 날린 적이 있습니다. 개발 DB인 줄 알고 UPDATE 쿼리를 실행했는데 운영 DB였습니다. 다행히 WHERE 절이 있어서 전체 데이터가 바뀌지는 않았지만, 한 50건 정도의 데이터가 잘못 수정되었습니다. 팀장님께 바로 보고드렸고, 백업 데이터로 복구했습니다. 그 이후로 쿼리 실행 전에 항상 DB 접속 정보를 확인하는 습관이 생겼습니다."

### 4차원 평가

| 차원 | 점수 (1-5) | 평가 |
|------|:----------:|------|
| 구체성 | 4 | "UPDATE 쿼리", "50건", "WHERE절" 등 구체적 상황 |
| 논리성 | 3 | 실수→보고→복구→교훈 흐름은 좋으나, 재발 방지 대책이 "확인 습관"으로 약함 |
| 진정성 | 5 | 운영 DB 실수를 솔직하게 인정. 매우 진정성 있음 |
| 직무관련성 | 4 | DB 운영 실수는 서버 개발자에게 공감되는 에피소드 |

**소계: 16/20**

### 피드백

**잘한 점**:
- 실수를 솔직하게 인정하고, 즉시 보고한 점이 좋음
- 구체적인 숫자(50건)와 맥락(WHERE절) 언급

**개선할 점**:
- "확인하는 습관이 생겼습니다"는 개인 수준의 해결책입니다. 면접관은 **시스템적 해결책**을 기대합니다.
- "그 이후 어떤 제도/프로세스를 만들었는가"가 빠져 있습니다.

**모범답안 방향**:

> "(동일 상황 설명 후) ... 이 사고 이후 세 가지 조치를 취했습니다. 첫째, 팀 내 DB 접속 가이드 문서를 작성해서 개발/스테이징/운영 DB 접속 정보를 색상으로 구분했습니다. 둘째, 운영 DB 접속 시 프롬프트에 빨간색 경고 배너가 뜨도록 .bashrc를 수정하는 방법을 팀에 공유했습니다. 셋째, DML 실행 전 반드시 SELECT로 영향 범위를 확인하는 '프리뷰 규칙'을 팀 코드 리뷰 체크리스트에 추가했습니다. **개인의 실수를 시스템으로 방지하는 것**이 중요하다는 교훈을 얻었습니다."

### 꼬리질문 예시

1. "만약 WHERE절이 없어서 전체 데이터가 변경되었다면 어떻게 했을 것 같나요?"
2. "팀장님의 반응은 어떠셨나요? 이후 신뢰 회복은 어떻게 하셨나요?"
3. "운영 환경 접근 권한 관리에 대해 어떤 생각을 가지고 계신가요?"

## Q4. 협업 스타일과 커뮤니케이션

### 면접관 질문

> "자소서에 기술 블로그를 40편 작성했다고 했는데, 팀에서 지식을 공유하는 것에 대해 어떤 철학을 가지고 계신가요? 구체적인 경험이 있으면 이야기해주세요."

### 김민수 시뮬레이션 답변

> "저는 배운 것을 글로 정리하면 이해가 더 깊어진다고 생각합니다. 그래서 Velog에 꾸준히 글을 쓰고 있습니다. 인턴 때도 Redis 캐싱을 적용한 경험을 블로그에 정리했는데, 댓글로 질문이 오면 답변해주면서 저도 더 공부가 되었습니다. 팀에서는... 사실 인턴이라 팀 내 지식 공유를 주도적으로 한 경험은 별로 없습니다. 하지만 네이버에 입사하면 팀 내 기술 공유 세션 같은 것에 적극 참여하고 싶습니다."

### 4차원 평가

| 차원 | 점수 (1-5) | 평가 |
|------|:----------:|------|
| 구체성 | 3 | 블로그 작성과 댓글 소통은 구체적이나, 팀 내 경험이 부재 |
| 논리성 | 3 | "글로 정리→이해 깊어짐"은 좋은 논리이나, 팀 차원 공유와 연결이 약함 |
| 진정성 | 5 | "인턴이라 주도적으로 한 경험은 별로 없다"는 매우 솔직한 답변 |
| 직무관련성 | 3 | 지식 공유는 중요하지만, 답변이 개인 활동에 머물러 있음 |

**소계: 14/20**

### 피드백

**잘한 점**:
- 경험이 없는 부분을 솔직하게 인정한 것은 진정성 면에서 높은 점수
- 블로그 댓글 소통 에피소드는 좋은 소재

**개선할 점**:
- "인턴이라 경험이 없다"에서 끝나면 안 됩니다. **작은 경험이라도 팀 지식 공유에 기여한 사례**를 찾아야 합니다.
- "참여하고 싶습니다"는 희망형 종결입니다.

**모범답안 방향**:

> "(블로그 이야기 후) ... 인턴 때 팀 내 온보딩 문서가 부족하다고 느꼈습니다. 신규 인턴이 저와 같은 삽질을 반복하지 않도록, 로컬 개발 환경 세팅 가이드와 자주 발생하는 에러 해결법을 Notion에 정리해서 팀 공유 스페이스에 올렸습니다. 다음 분기에 들어온 인턴이 '이 문서 덕분에 세팅 시간이 반나절에서 2시간으로 줄었다'고 말해줬을 때, **문서화의 힘**을 실감했습니다. 네이버에서도 D2 기술 블로그 기고를 통해 사내외 지식 공유에 기여하는 것을 1년차 목표로 설정하고 있습니다."

### 꼬리질문 예시

1. "블로그에 가장 반응이 좋았던 글은 어떤 것이고, 왜 반응이 좋았다고 생각하나요?"
2. "기술 블로그 글을 쓸 때 어떤 프로세스를 따르나요? 초안부터 발행까지의 과정을 설명해주세요."
3. "팀원이 잘못된 기술적 의견을 고집할 때, 어떻게 설득하시겠어요?"

## Q5. 지원동기와 성장 비전

### 면접관 질문

> "마지막 질문입니다. 왜 하필 네이버이고, 왜 하필 Search Platform 팀인가요? 다른 대기업이나 스타트업이 아닌 이유를 이야기해주세요."

### 김민수 시뮬레이션 답변

> "네이버는 제가 가장 많이 사용하는 서비스이기도 하고, 기술력이 뛰어난 회사라고 생각합니다. Search Platform은 검색이 네이버의 핵심이니까, 가장 기술적으로 도전적인 환경일 것 같아서 지원했습니다. 저는 인턴에서 Redis 캐싱이나 쿼리 최적화를 경험하면서 성능 최적화에 관심을 갖게 되었는데, 검색 플랫폼이야말로 성능이 가장 중요한 분야라고 생각합니다. 입사하면 대규모 시스템에서 경험을 쌓아 성장하고 싶습니다."

### 4차원 평가

| 차원 | 점수 (1-5) | 평가 |
|------|:----------:|------|
| 구체성 | 2 | "가장 많이 사용", "기술력이 뛰어난" — 모든 지원자가 하는 말 |
| 논리성 | 3 | "성능 최적화 관심→검색 플랫폼" 연결은 있으나, 왜 카카오/쿠팡이 아닌지 설명 부족 |
| 진정성 | 3 | 무난하지만 특별한 진정성은 느껴지지 않음 |
| 직무관련성 | 3 | 검색 플랫폼 언급은 했으나, 해당 팀의 구체적 기술/프로젝트 언급 없음 |

**소계: 11/20**

### 피드백

**잘한 점**:
- 인턴 경험 → 검색 플랫폼으로의 연결고리 시도

**개선할 점**:
