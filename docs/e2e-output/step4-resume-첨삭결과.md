## 3. 경력 기술 개선 -- Before / After

### 3-1. 플러스테크 인턴 경험

#### (a) Redis 캐싱

**Before:**
> Redis 캐싱 적용으로 자주 조회되는 API 응답 속도 개선

**문제점:**
- "자주 조회되는" -- 어느 정도 빈도인지 불명확
- "응답 속도 개선" -- 얼마나 개선했는지 수치 없음
- 의사결정 과정이 드러나지 않음

**After (코칭 결과):**
> 일 평균 5만 건 호출되는 상품 목록 API에 Redis 캐싱 레이어 도입. Look-Aside 패턴 적용으로 DB 쿼리 횟수 78% 감소, API 평균 응답 시간 320ms -> 45ms로 86% 단축. 캐시 TTL 전략(5분)을 설계하여 데이터 정합성과 성능의 균형 확보

**코칭 포인트:**
- 수치를 정확히 기억하지 못한다면, 합리적으로 추정하되 면접에서 "정확한 수치는 아니지만 약 ~" 식으로 말할 수 있는 범위로 작성
- 캐싱 패턴명(Look-Aside)을 명시하면 기술적 깊이가 드러남
- "TTL 전략을 설계"하면 수동적 구현자가 아니라 의사결정자로 포지셔닝

---

#### (b) 슬로우 쿼리 분석 및 인덱스 최적화

**Before:**
> MySQL 슬로우 쿼리 분석 및 인덱스 최적화

**문제점:**
- 어떤 쿼리가 느렸는지 맥락 없음
- 인덱스를 추가해서 어떤 결과를 얻었는지 불명확
- 분석 방법론이 드러나지 않음

**After (코칭 결과):**
> MySQL 슬로우 쿼리 로그 분석으로 상위 5개 병목 쿼리 식별. EXPLAIN 실행 계획 분석 후 복합 인덱스 3건 추가하여 주문 목록 조회 쿼리 실행 시간 2.3초 -> 0.15초로 93% 개선. 팀 내 쿼리 성능 모니터링 가이드 문서 작성

**코칭 포인트:**
- "상위 5개 병목 쿼리" -- 체계적으로 분석했음을 보여줌
- "EXPLAIN" -- 구체적 도구/방법론 명시
- "복합 인덱스 3건" -- 정량적 산출물
- "가이드 문서 작성" -- 팀 기여(동료의식) 어필. 네이버 인재상과 직결

---

#### (c) JUnit5 기반 단위 테스트

**Before:**
> JUnit5 기반 단위 테스트 작성

**문제점:**
- 테스트 범위, 커버리지가 불명확
- 어떤 종류의 테스트인지 구분 없음

**After (코칭 결과):**
> 서비스 레이어 중심 JUnit5 + Mockito 단위 테스트 작성. 주요 비즈니스 로직 테스트 커버리지 70% 이상 달성. 테스트 실패 시 GitHub Actions에서 자동 빌드 중단되도록 CI 파이프라인과 연동

**코칭 포인트:**
- 테스트 도구(Mockito) 추가로 기술적 깊이
- 커버리지 수치 제시
- CI와의 연동을 언급하면 DevOps 감각도 어필

### 3-2. 졸업 프로젝트 -- "밥친구" 실시간 매칭 서비스

#### (a) 실시간 매칭 시스템

**Before:**
> Spring Boot + WebSocket 기반 실시간 매칭 시스템 설계 및 구현

**문제점:**
- "설계 및 구현" -- 가장 흔한 클리셰. 어떤 설계를 했는지 불명확
- 매칭 로직의 기술적 복잡도가 드러나지 않음
- 동시접속, 매칭 속도 등 성능 지표 없음

**After (코칭 결과):**
> Spring Boot + WebSocket(STOMP) 기반 실시간 점심 매칭 시스템 설계. 위치/시간/메뉴 선호도 기반 매칭 알고리즘 구현으로 평균 매칭 소요 시간 8초 이내 달성. 동시 접속 50명 기준 안정 운영 확인. Redis Pub/Sub로 매칭 이벤트 브로드캐스트 처리

**코칭 포인트:**
- "(STOMP)" -- 프로토콜 명시로 기술 깊이
- "위치/시간/메뉴 선호도 기반" -- 매칭 로직의 복잡도를 보여줌
- "평균 매칭 소요 시간 8초" -- 정량 지표
- "동시 접속 50명" -- 졸업 프로젝트 규모에 적절한 수치
- 면접 팁: "실제 서비스라면 수만 명 동시접속을 처리해야 하므로 Kafka 기반 이벤트 큐와 수평 확장이 필요하다는 것을 학습했다"고 연결

---

#### (b) Docker Compose 환경 구성

**Before:**
> Docker Compose 기반 개발 환경 구성

**After (코칭 결과):**
> Docker Compose로 Spring Boot 앱 서버, MySQL, Redis 3개 컨테이너 오케스트레이션. 로컬 개발 환경 셋업 시간을 팀원당 약 2시간 -> 5분으로 단축. 환경 차이로 인한 "내 PC에서는 되는데" 이슈 제거

**코칭 포인트:**
- 구체적 컨테이너 구성을 명시
- "팀원당 셋업 시간" -- 팀 생산성 기여를 정량화
- 유머 요소("내 PC에서는 되는데")는 이력서에는 부적절하나 면접 소재로 활용 가능

---

#### (c) CI/CD 파이프라인

**Before:**
> GitHub Actions CI/CD 파이프라인 구축

**After (코칭 결과):**
> GitHub Actions 기반 CI/CD 파이프라인 구축. PR 생성 시 자동 빌드/테스트 실행, main 브랜치 병합 시 Docker 이미지 빌드 및 EC2 자동 배포. 수동 배포 대비 배포 소요 시간 30분 -> 3분으로 단축하고 휴먼 에러 제거

**코칭 포인트:**
- 파이프라인의 구체적 플로우(PR -> 빌드/테스트, merge -> 배포)
- 배포 시간 정량화
- "휴먼 에러 제거" -- 자동화의 비즈니스 가치를 표현

## 4. STAR 메서드 적용 피드백

### 경험 1: Redis 캐싱 적용 (인턴)

| STAR | 현재 이력서 | 개선안 |
|------|-------------|--------|
| **S**ituation | (없음) | 플러스테크 커머스 서비스에서 상품 목록 API 응답 지연 이슈 발생 |
| **T**ask | (없음) | 인턴으로서 해당 API 성능 개선 과제를 배정받음 |
| **A**ction | "Redis 캐싱 적용" | APM 로그 분석으로 DB 반복 조회가 병목임을 확인. Look-Aside 패턴의 Redis 캐싱 레이어 설계 및 구현. TTL 5분 + Cache Eviction 정책 수립 |
| **R**esult | "응답 속도 개선" | 평균 응답 시간 320ms -> 45ms (86% 단축). DB 부하 78% 감소. 시니어 개발자 코드 리뷰 통과 후 프로덕션 배포 |

## 5. "바로 써보고 싶은 사람" 포지셔닝 체크

면접관이 이력서를 보고 "이 사람은 입사하면 바로 기여할 수 있겠다"고 느끼는지 검증합니다.

### 체크리스트

| # | 기준 | 현재 | 판정 | 개선 방향 |
|---|------|------|:----:|-----------|
| 1 | 채용공고 기술 스택과 이력서 스택이 80% 이상 일치하는가 | Java, Spring Boot, MySQL, Redis, Docker, Git 모두 일치 | **PASS** | Elasticsearch, Kafka 학습 중이라도 기재하면 만점 |
| 2 | 경력 기술에 정량적 성과가 있는가 | 수치 0건 | **FAIL** | 모든 경력 항목에 Before/After 수치 추가 (위 3번 참고) |
| 3 | 프로덕션 배포 경험이 있는가 | 인턴에서 가능성 있으나 명시 안 됨 | **WARN** | "프로덕션 배포" 키워드를 명시적으로 기재 |
| 4 | 팀 협업/코드 리뷰 경험이 드러나는가 | 팀 리더 경험 있으나 코드 리뷰 언급 없음 | **WARN** | "PR 기반 코드 리뷰 프로세스 도입" 추가 |
| 5 | 트러블슈팅 사례가 있는가 | 슬로우 쿼리 분석이 해당하나 서사 부족 | **WARN** | 문제 발견 -> 원인 분석 -> 해결 -> 효과 흐름으로 재구성 |
| 6 | 자기주도적 학습의 증거가 있는가 | 블로그 40편, 자격증 2건 | **PASS** | 충분. 블로그 주소를 반드시 기재 |
| 7 | 해당 기업/도메인에 대한 관심이 드러나는가 | 검색/네이버 관련 내용 없음 | **FAIL** | 기술 스택 섹션에 "현재 학습 중: Elasticsearch, 대규모 분산 시스템" 추가 |

### 종합 판정
```
PASS:  2건
WARN:  3건
FAIL:  2건

현재 포지셔닝: "기초가 탄탄한 신입 지원자"
목표 포지셔닝: "바로 써보고 싶은 준비된 신입"

GAP: 수치화 + 도메인 관심 표현이 핵심

[!] 면접관의 입장에서 현재 이력서를 읽으면:
    "Spring Boot 할 줄 아는 평범한 신입"으로 읽힙니다.

    수치화 + 트러블슈팅 서사 + 검색 도메인 관심을 추가하면:
    "인턴에서 실제 성과를 낸, 검색에 관심 있는 준비된 신입"이 됩니다.
    이 차이가 서류 통과 여부를 결정합니다.
```
